\section{\texorpdfstring{\emph{Abstract}}{Abstract}}\label{abstract}

This specification defines three Verifiable Random Function with
Additional Data (VRF-AD) schemes -- IETF VRF, Thin VRF, and Pedersen VRF
-- built on a transcript-based Fiat-Shamir transform with support for
multiple input/output pairs via delinearization. The IETF VRF extends
\href{https://datatracker.ietf.org/doc/rfc9381}{RFC-9381} {[}1{]}; the
Thin VRF and Pedersen VRF follow the constructions introduced by
\href{https://eprint.iacr.org/2023/002}{BCHSV23} {[}2{]}, with the
Pedersen VRF serving as a building block for anonymized ring signatures
as described in \href{https://github.com/davxy/ring-proof-spec}{VG24}
{[}3{]}. All schemes are instantiated over the Bandersnatch elliptic
curve, constructed over the BLS12-381 scalar field as specified in
\href{https://eprint.iacr.org/2021/1152}{MSZ21} {[}4{]}.

\section{1. Preliminaries}\label{preliminaries}

\subsection{1.1. Groups and Fields}\label{groups-and-fields}

\begin{itemize}
\tightlist
\item
  \(\bold{G}\): Bandersnatch curve cyclic group of prime order \(r\),
  defined over the base field of prime order \(q\).
\item
  \(\bold{F}\): Scalar field of prime order \(r\)
  (i.e.~\(\mathbb{Z}_r\)).
\item
  \(\bold{\Sigma}^k\): Octet strings with length \(k \in \mathbb{N}\)
  (\(*\) for arbitrary length).
\item
  \(\mathcal{O}\): Identity point of \(\bold{G}\).
\end{itemize}

The EC group \(\bold{G}\) is the prime subgroup of the Bandersnatch
elliptic curve, in Twisted Edwards form, with finite field and curve
parameters as specified in
\href{https://eprint.iacr.org/2021/1152}{MSZ21} {[}4{]}. For this group,
\texttt{fLen} = \texttt{qLen} = \(32\) and \texttt{cofactor} = \(4\).

All point arithmetic MUST be performed in \(\bold{G}\). Points not in
\(\bold{G}\) MUST be rejected at all entry points; accepting a point on
the full curve but outside the prime-order subgroup enables
small-subgroup attacks that break the VRF relation.

\begin{itemize}
\tightlist
\item

\item
  \(Y \in \bold{G}\): Public key point defined as \(x \cdot G\).
\item
  \(i \in \bold{\Sigma}^*\): VRF input data.
\item
  \(I \in \bold{G}\): VRF input point.
\item

\item
  \(o \in \bold{\Sigma}^k\): VRF output hash.
\item
  \(T\): Transcript state.
\end{itemize}

\subsection{1.3. VRF-AD}\label{vrf-ad}

Regardless of the specific scheme, a \emph{Verifiable Random Function
with Additional Data (VRF-AD)} can be concisely represented by three
primary functions:

\textbf{Abstract interface}:

\begin{itemize}
\tightlist
\item
  \(\texttt{prove}(x \in \bold{F}, \overline{io} \in (\bold{G}\times \bold{G})^n, ad \in \bold{\Sigma}^*) \to \Pi\)
\item
  \(\texttt{verify}(Y \in \bold{G}, \overline{io} \in (\bold{G}\times \bold{G})^n, ad \in \bold{\Sigma}^*, \Pi) \to (\top \mid \bot)\)
\item
  \(\texttt{output}(O \in \bold{G}) \to o \in \bold{\Sigma}^N\)
\end{itemize}

For Pedersen VRF (section 4), the public key \(Y\) is not an explicit
input to \(\texttt{verify}\); the proof \(\Pi\) carries a blinded
commitment \(\bar{Y}\) instead.

The additional data \(ad\) is an arbitrary-length octet-string signed
together with the VRF output. It does not influence the produced VRF
output. The length of \(ad\) MUST NOT exceed \(2^{32} - 1\) bytes, as
the length is encoded via \(\texttt{enc\_32}\) in the transcript
(section 1.6.1).

\subsection{1.4. Constants}\label{constants}

\begin{itemize}
\tightlist
\item
  \texttt{suite\_id} = \texttt{{[}0x01,\ 0x01,\ 0x01,\ 0x01{]}} --- a
  4-byte fixed-width identifier encoding the protocol profile:
\end{itemize}

\begin{longtable}[]{@{}llll@{}}
\toprule\noalign{}
Name & Byte & Value & Meaning \\
\midrule\noalign{}
\endhead
\bottomrule\noalign{}
\endlastfoot
version & 0 & 0x01 & Suite version (v1) \\
curve & 1 & 0x01 & Bandersnatch \\
hash & 2 & 0x01 & SHA-512 \\

\end{longtable}

The version byte bundles several tightly-coupled choices: transcript
construction (HashTranscript), nonce algorithm (RFC-8032), challenge

these changes.

\begin{itemize}
\tightlist
\item
  \texttt{challenge\_len} = 16 bytes (128-bit security).
\item
  \texttt{expanded\_scalar\_len} =
  \(\lceil(\lceil\log_2(r)\rceil + 128) / 8\rceil\) = 48 bytes.
\end{itemize}

Domain separation tags used throughout the protocol:

\subsection{1.5. Codec}\label{codec}

\begin{itemize}
\tightlist
\item
  \(\texttt{enc\_scalar}(s \in \bold{F})\): Encodes a scalar into 32
  octets in little-endian representation.
\item
  \(\texttt{dec\_scalar}(buf \in \bold{\Sigma}^{32})\): Interpret octet
  string \(buf\) as a little-endian integer. MUST output ``INVALID'' if
  the resulting value is \(\geq r\).
\item
  \(\texttt{enc\_point}(P \in \bold{G})\): Encodes a point in compressed
  form. The \(y\) coordinate is serialized in little-endian and the most
  significant bit of the last octet encodes the sign of the
  \(x\)-coordinate. This gives \texttt{ptLen} = \texttt{fLen} = \(32\).
\item
  \(\texttt{dec\_point}(buf \in \bold{\Sigma}^{32})\): Interpret octet
  string \(buf\) as a compressed point. Mask the sign bit from the last
  octet and interpret the result as a little-endian integer. MUST output
  ``INVALID'' if the resulting value is \(\geq q\). Otherwise,
  decompress the point and MUST output ``INVALID'' if it does not decode
  to a point on the prime subgroup \(\bold{G}\).
\item
  \(\texttt{dec\_scalar\_mod}(buf \in \bold{\Sigma}^*)\): Interpret
  octet string \(buf\) as a little-endian integer and reduce modulo the
  prime field order \(r\).
\item
  \(\texttt{enc\_32}(n \in \mathbb{N}_{2^{32}})\): Encode integer \(n\)
  as a 4-byte little-endian octet string.
\end{itemize}

Aggregate types (e.g.~proofs) MUST be encoded as the concatenation of
their individual fields in the order given by the structure definition,
without any separator.

\subsection{1.6. Procedures}\label{procedures}

\subsubsection{1.6.1. Transcript}\label{transcript}

The transcript provides a Fiat-Shamir transform with an absorb/squeeze
interface. Data is absorbed into an internal hash state; output bytes
are squeezed from it. After the first squeeze, \(\texttt{absorb}\) MUST
NOT be called.

\textbf{Abstract interface}:

\begin{itemize}
\tightlist
\item
  \(\texttt{new\_transcript}() \to T\): Create a fresh transcript
  instance and absorb \(\texttt{suite\_id}\).
\item
  \(\texttt{absorb}(data \in \bold{\Sigma}^*)\): Feed bytes into the
  hash state. MUST NOT be called after squeeze.
\item
  \(\texttt{squeeze}(n \in \mathbb{N}) \to \bold{\Sigma}^n\): Produce
  \(n\) output bytes.
\item
  \(\texttt{fork}() \to T\): Clone the transcript state.
\end{itemize}

A concrete instantiation
(\texttt{HashTranscript\textless{}SHA-512\textgreater{}}) is given in
Appendix B.

\subsubsection{1.6.2. VRF Input}\label{vrf-input}

The VRF input point \(I \in \bold{G}\) is derived from the input
octet-string using the \(\texttt{hash\_to\_curve}\) method defined in
section 3 of \href{https://datatracker.ietf.org/doc/rfc9380}{RFC-9380}
{[}5{]}, instantiated with the \emph{Elligator 2} map to curve (section
6.8.2) and \(\texttt{expand\_message\_xmd}\) with SHA-512 (section
5.3.1).

This is the random oracle (\texttt{\_RO\_}) construction: the input is
hashed to two independent field elements, each is mapped to a curve
point via Elligator 2, and the results are added.

\[I \gets \texttt{hash\_to\_curve}(DST, i)\]

The domain separation tag is:

\[DST = \text{"ECVRF\_"} \;\Vert\; \texttt{h2c\_suite\_id} \;\Vert\; \texttt{suite\_id}\]

where \texttt{h2c\_suite\_id} =
\texttt{"Bandersnatch\_XMD:SHA-512\_ELL2\_RO\_"} is the RFC-9380 suite
identifier string determined by the curve, hash, and h2c bytes of
\texttt{suite\_id}.

Verifiers MUST independently compute each \(I_i\) from the corresponding
input octet-string using the procedure above. Accepting prover-supplied
input points without recomputation breaks the VRF security guarantees,
and in the case of Thin VRF (section 3), enables trivial forgery.

\subsubsection{1.6.3. VRF Output}\label{vrf-output}

The VRF output point is generated from the VRF input point and secret
key scalar:

\[O \gets x \cdot I\]

The VRF output hash is a fixed-length octet string derived from the
output point using a transcript-based point-to-hash procedure. The
procedure is deliberately independent of the proof scheme: for a given
key and input, the output point \(O = x \cdot I\) is unique regardless
of whether IETF VRF, Thin VRF, or Pedersen VRF is used to prove
correctness. The scheme determines how the proof is constructed, not the
VRF output itself. This separation allows applications to obtain
consistent output hashes across schemes for the same underlying
evaluation.

\textbf{Input}:

\begin{itemize}
\tightlist
\item
  \(O \in \bold{G}\): VRF output point.
\item
  \(N \in \mathbb{N}\): Desired output length in bytes. MUST be fixed
  per application context; \(N\) is not absorbed into the transcript, so
  \(\texttt{squeeze}(N_1)\) is a prefix of \(\texttt{squeeze}(N_2)\) for
  \(N_1 < N_2\).
\end{itemize}

\textbf{Output}:

\begin{itemize}
\tightlist
\item
  \(o \in \bold{\Sigma}^N\): VRF output hash.
\end{itemize}

\textbf{Steps}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \(T \gets \texttt{new\_transcript}()\)
\item
  \(T.\texttt{absorb}(\texttt{PointToHash})\)
\item
  \(T.\texttt{absorb}(\texttt{enc\_point}(O))\)
\item
  \(o \gets T.\texttt{squeeze}(N)\)
\end{enumerate}

\subsubsection{1.6.4. Delinearization}\label{delinearization}

When multiple input/output pairs are provided (\(n \geq 2\)), they are
merged into a single pair using delinearization scalars derived from the
transcript. This prevents an attacker from mixing components across
pairs.

\textbf{Input}:

\begin{itemize}
\tightlist
\item
  \(\overline{io} \in (\bold{G}\times \bold{G})^n\): Sequence of
  input/output pairs.
\item
  \(T\): Transcript state.
\end{itemize}

\textbf{Output}:

\begin{itemize}
\tightlist
\item
  \((I_m, O_m) \in \bold{G}\times \bold{G}\): Merged input/output pair.
\end{itemize}

\textbf{Steps}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  If \(n = 0\): return \((\mathcal{O}, \mathcal{O})\)
\item
  If \(n = 1\): return \((I_0, O_0)\)
\item
  \(T.\texttt{absorb}(\texttt{Delinearize})\)
\item
  For \(i = 0, \ldots, n-1\):
  \(z_i \gets \texttt{dec\_scalar\_mod}(T.\texttt{squeeze}(\texttt{challenge\_len}))\)
\item
  \(I_m \gets \sum_{i=0}^{n-1} z_i \cdot I_i,\ O_m \gets \sum_{i=0}^{n-1} z_i \cdot O_i\)
\end{enumerate}

\subsubsection{1.6.5. VRF Transcript}\label{vrf-transcript}

Shared transcript construction used by all VRF-AD schemes. Absorbs
input/output pairs, merges them via delinearization (section 1.6.4), and
absorbs additional data.

\textbf{Input}:

\begin{itemize}
\tightlist
\item
  \(scheme\): Scheme identifier tag.
\item
  \(\overline{io} \in (\bold{G}\times \bold{G})^n\): Sequence of
  input/output pairs.
\item
  \(ad \in \bold{\Sigma}^*\): Additional data octet-string.
\end{itemize}

\textbf{Output}:

\begin{itemize}
\tightlist
\item
  \(T\): Transcript state (with \(ad\) absorbed).
\item
  \((I_m, O_m) \in \bold{G}\times \bold{G}\): Merged input/output pair.
\end{itemize}

\textbf{Steps}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \(T \gets \texttt{new\_transcript}()\)
\item
  \(T.\texttt{absorb}(scheme || \texttt{enc\_32}(n))\)
\item
  For each \((I_i, O_i)\) in \(\overline{io}\):
  \(T.\texttt{absorb}(\texttt{enc\_point}(I_i) \;\Vert\; \texttt{enc\_point}(O_i))\)
\item
  \((I_m, O_m) \gets \texttt{delinearize}(\overline{io}, T.\texttt{fork}())\)
\item
  \(T.\texttt{absorb}(\texttt{enc\_32}(\texttt{len}(ad)) \;\Vert\; ad)\)
\item
  Return \((T, (I_m, O_m))\)
\end{enumerate}

\subsubsection{1.6.6. Nonce}\label{nonce}

Deterministic nonce generation inspired by {[}RFC-8032{]} section 5.1.6.
The transcript carries shared state from \(\texttt{vrf\_transcript}\),
binding the nonce to the I/O pairs and additional data.

\textbf{Input}:

\begin{itemize}
\tightlist
\item
  \(d \in \bold{F}\): Secret scalar.
\item
  \(T\): Transcript state (consumed).
\end{itemize}

\textbf{Output}:

\begin{itemize}
\tightlist
\item
  \(k \in \bold{F}\): Nonce scalar.
\end{itemize}

\textbf{Steps}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \(T' \gets T.\texttt{fork}()\)
\item
  \(T'.\texttt{absorb}(\texttt{NonceExpand} \;\Vert\; \texttt{enc\_scalar}(d))\)
\item
  \(h \gets T'.\texttt{squeeze}(64)\)
\item
  \(T.\texttt{absorb}(\texttt{Nonce} \;\Vert\; h)\)
\item
  \(k \gets \texttt{dec\_scalar\_mod}(T.\texttt{squeeze}(\text{expanded\_scalar\_len}))\)
\item
  If \(k = 0\): abort (implementation error; probability
  \(\approx 2^{-253}\)).
\end{enumerate}

Note: \(T\) is consumed (mutated then squeezed). Callers must pass forks
where the transcript is needed afterwards.

\subsubsection{1.6.7. Challenge}\label{challenge}

Derives a challenge scalar by absorbing curve points into the transcript
and squeezing.

\textbf{Input}:

\begin{itemize}
\tightlist
\item
  \(\bar{P} \in \bold{G}^m\): Sequence of \(m\) points.
\item
  \(T\): Transcript state (consumed).
\end{itemize}

\textbf{Output}:

\begin{itemize}
\tightlist
\item
  \(c \in \bold{F}\): Challenge scalar.
\end{itemize}

\textbf{Steps}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \(T.\texttt{absorb}(\texttt{Challenge})\)
\item
  For each \(P_i\) in \(\bar{P}\):
  \(T.\texttt{absorb}(\texttt{enc\_point}(P_i))\)
\item
  \(c \gets \texttt{dec\_scalar\_mod}(T.\texttt{squeeze}(\texttt{challenge\_len}))\)
\end{enumerate}

\section{2. IETF VRF}\label{ietf-vrf}

Based on IETF \href{https://datatracker.ietf.org/doc/rfc9381}{RFC-9381}
which is extended with a transcript-based Fiat-Shamir transform, support
for additional data (\(ad\)), and multiple I/O pairs via
delinearization. These changes make this scheme incompatible with
standard RFC-9381 implementations and test vectors.

\subsection{2.1. Prove}\label{prove}

\textbf{Input}:

\begin{itemize}
\tightlist
\item
  \(x \in \bold{F}\): Secret key.
\item
  \(\overline{io} \in (\bold{G}\times \bold{G})^n\): VRF input/output
  pairs.
\item
  \(ad \in \bold{\Sigma}^*\): Additional data octet-string.
\end{itemize}

\textbf{Output}:

\begin{itemize}
\tightlist
\item
  \(\pi = (c, s) \in (\bold{F}, \bold{F})\): Schnorr-like proof.
\end{itemize}

\textbf{Steps}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \(Y \gets x \cdot G\)
\item
  \((T, (I_m, O_m)) \gets \texttt{vrf\_transcript}(\texttt{IetfVrf}, \overline{io}, ad)\)
\item
  \(k \gets \texttt{nonce}(x, T.\texttt{fork}())\)
\item
  \(U \gets k \cdot G\), \(\quad V \gets k \cdot I_m\)
\item
  \(c \gets \texttt{challenge}([Y, U, V], T)\)
\item
  \(s \gets k + c \cdot x\)
\item
  \(\pi \gets (c, s)\)
\end{enumerate}

\subsection{2.2. Verify}\label{verify}

\textbf{Input}:

\begin{itemize}
\tightlist
\item
  \(Y \in \bold{G}\): Public key.
\item
  \(\overline{io} \in (\bold{G}\times \bold{G})^n\): VRF input/output
  pairs.
\item
  \(ad \in \bold{\Sigma}^*\): Additional data octet-string.
\item
  \(\pi = (c, s) \in (\bold{F}, \bold{F})\): Schnorr-like proof.
\end{itemize}

\textbf{Output}:

\begin{itemize}
\tightlist
\item
  \(\theta \in \{ \top, \bot \}\): \(\top\) if proof is valid, \(\bot\)
  otherwise.
\end{itemize}

\textbf{Steps}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Validate \(Y\) and all \(I_i, O_i\)
  \(\in \bold{G}\setminus \{\mathcal{O}\}\), output \(\bot\) if any is
  invalid or the identity.
\item
  \((T, (I_m, O_m)) \gets \texttt{vrf\_transcript}(\texttt{IetfVrf}, \overline{io}, ad)\)
\item
  \(U \gets s \cdot G - c \cdot Y\)
\item
  \(V \gets s \cdot I_m - c \cdot O_m\)
\item
  \(c' \gets \texttt{challenge}([Y, U, V], T)\)
\item
  \(\theta \gets \top \text{ if } c = c' \text{ else } \bot\)
\end{enumerate}

\section{3. Thin VRF}\label{thin-vrf}

Thin VRF is derived from the PedVRF construction in section 4 of
\href{https://eprint.iacr.org/2023/002}{BCHSV23} {[}2{]} by removing the
blinding mechanism entirely (see remark on page 13 of the paper).
Without blinding, Pedersen VRF reduces to two independent DLEQ checks on
\((G, Y)\) and \((I_m, O_m)\) with the same secret \(x\). Thin VRF
merges these into a single DLEQ relation by prepending \((G, Y)\) to the
I/O pairs and applying delinearization, then proves it with a
Schnorr-like proof \((R, s)\). Storing the nonce commitment \(R\)
(rather than the challenge \(c\)) enables batch verification.

\textbf{Security}: VRF input points MUST be constructed via
hash-to-curve. If a prover knows \(d\) such that \(I = d \cdot G\), they
can forge arbitrary outputs for that input, because the delinearization
merges the Schnorr and VRF pairs into a single check that collapses when
all points are multiples of \(G\).

\tightlist

\item

\item

\item

\item

\end{enumerate}

\textbf{Input}:

\begin{itemize}
\tightlist

\item

\item

\begin{itemize}
\tightlist

Multiple Thin VRF proofs can be verified together by combining the

\tightlist
\item
  For \(j = 0, \ldots, N-1\): a tuple
  \((Y_j, \overline{io}_j, ad_j, \pi_j)\) where:

  \tightlist

    \(Y_j \in \bold{G}\): Public key.
  \item
    \(\overline{io}_j \in (\bold{G}\times \bold{G})^{M_j}\): VRF

  \item
    \(ad_j \in \bold{\Sigma}^*\): Additional data octet-string.
  \item
    \(\pi_j = (R_j, s_j) \in (\bold{G}, \bold{F})\): Thin VRF proof.
  \end{itemize}
\end{itemize}

\textbf{Output}:

\begin{itemize}
\tightlist
\item
  \(\theta \in \{ \top, \bot \}\): \(\top\) if all proofs verify,
  \(\bot\) otherwise.
\end{itemize}

\textbf{Steps}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  For each proof \(j\):

  \begin{enumerate}
  \def\labelenumii{\alph{enumii}.}
  \tightlist
  \item
    Validate \(Y_j\), \(R_j\), and all \(I_{j,i}, O_{j,i}\)
    \(\in \bold{G}\setminus \{\mathcal{O}\}\), output \(\bot\) if any is
    invalid or the identity.
  \item
    \((T_j, (I_{m,j}, O_{m,j})) \gets \texttt{vrf\_transcript}(\texttt{ThinVrf}, [(G, Y_j)] \;\Vert\; \overline{io}_j, ad_j)\)
  \item
    \(c_j \gets \texttt{challenge}([R_j], T_j)\)
  \end{enumerate}
\item
  Derive random weights:

  \begin{enumerate}
  \def\labelenumii{\alph{enumii}.}
  \tightlist
  \item
    \(T_w \gets \texttt{new\_transcript}()\)
  \item
    \(T_w.\texttt{absorb}(\texttt{"thin-batch"})\)
  \item
    For each \(j\):
    \(T_w.\texttt{absorb}(\texttt{enc\_scalar}(c_j) \;\Vert\; \texttt{enc\_scalar}(s_j))\)
  \end{enumerate}
\item
  Check the combined equation:
  \[\sum_{j=0}^{N-1} w_j \cdot (s_j \cdot I_{m,j} - R_j - c_j \cdot O_{m,j}) = \mathcal{O}\]
  where
  \(w_j \gets \texttt{dec\_scalar\_mod}(T_w.\texttt{squeeze}(\texttt{challenge\_len}))\).
\end{enumerate}

\section{4. Pedersen VRF}\label{pedersen-vrf}

Pedersen VRF resembles IETF EC-VRF but replaces the public key with a
Pedersen commitment to the secret key, which makes this VRF useful in
anonymized ring proofs.

The scheme proves that the output has been generated with a secret key
associated with a blinded public key (instead of the public key). The
blinded public key is a cryptographic commitment to the public key, and
it can be unblinded to prove that the output of the VRF corresponds to
the public key of the signer.

This specification mostly follows the design proposed by
\href{https://eprint.iacr.org/2023/002}{BCHSV23} {[}2{]} in section 4
with some details about blinding base point value and challenge
generation procedure.

The \emph{blinding base} \(B \in \bold{G}\) is defined as:
\[\footnotesize\begin{aligned}
x &= 5226425992571220769365843487102064307101272980791993134273780736997544949382 \\
y &= 46544868206883149332782258938702216106598247683423727002885664111567608220426
\end{aligned}\]

A point with unknown discrete logarithm derived using the
\texttt{hash\_to\_curve} function as described in section 1.6.2 with
input the string: \texttt{"pedersen-blinding"}.

\subsection{4.1. Prove}\label{prove-2}

\textbf{Input}:

\begin{itemize}
\tightlist
\item
  \(x \in \bold{F}\): Secret key.
\item
  \(\overline{io} \in (\bold{G}\times \bold{G})^n\): VRF input/output
  pairs.
\item
  \(ad \in \bold{\Sigma}^*\): Additional data octet-string.
\end{itemize}

\textbf{Output}:

\begin{itemize}
\tightlist
\item
  \(\pi = (\bar{Y}, R, O_k, s, s_b) \in (\bold{G}, \bold{G}, \bold{G}, \bold{F}, \bold{F})\):
  Pedersen proof.
\item
  \(b \in \bold{F}\): Blinding factor.
\end{itemize}

\textbf{Steps}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \((T, (I_m, O_m)) \gets \texttt{vrf\_transcript}(\texttt{PedersenVrf}, \overline{io}, ad)\)
\item
  \(b \gets \texttt{blinding}(x, T.\texttt{fork}())\) (see Appendix A.2)
\item
  \(\bar{Y} \gets x \cdot G + b \cdot B\)
\item
  \(T_k \gets T.\texttt{fork}()\),
  \(\quad T_k.\texttt{absorb}(\texttt{enc\_scalar}(b))\),
  \(\quad k \gets \texttt{nonce}(x, T_k)\)
\item
  \(T_{kb} \gets T.\texttt{fork}()\),
  \(\quad T_{kb}.\texttt{absorb}(\texttt{enc\_scalar}(x))\),
  \(\quad k_b \gets \texttt{nonce}(b, T_{kb})\)
\item
  \(R \gets k \cdot G + k_b \cdot B\)
\item
  \(O_k \gets k \cdot I_m\)
\item
  \(c \gets \texttt{challenge}([\bar{Y}, R, O_k], T)\)
\item
  \(s \gets k + c \cdot x\)
\item
  \(s_b \gets k_b + c \cdot b\)
\item
  \(\pi \gets (\bar{Y}, R, O_k, s, s_b)\)
\end{enumerate}

The nonce cross-binding is critical: \(k\) is bound to \(b\) (step 4)
and \(k_b\) is bound to \(x\) (step 5). This prevents secret/blinding
recovery from two proofs with the same (secret, input, ad) but different
blinding factors.

\subsection{4.2. Verify}\label{verify-2}

\textbf{Input}:

\begin{itemize}
\tightlist
\item
  \(\overline{io} \in (\bold{G}\times \bold{G})^n\): VRF input/output
  pairs.
\item
  \(ad \in \bold{\Sigma}^*\): Additional data octet-string.
\item
  \(\pi = (\bar{Y}, R, O_k, s, s_b) \in (\bold{G}, \bold{G}, \bold{G}, \bold{F}, \bold{F})\):
  Pedersen proof.
\end{itemize}

\textbf{Output}:

\begin{itemize}
\tightlist
\item
  \(\theta \in \{ \top, \bot \}\): \(\top\) if proof is valid, \(\bot\)
  otherwise.
\end{itemize}

\textbf{Steps}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Validate \(\bar{Y}\), \(R\), \(O_k\), and all \(I_i, O_i\)
  \(\in \bold{G}\setminus \{\mathcal{O}\}\), output \(\bot\) if any is
  invalid or the identity.
\item
  \((T, (I_m, O_m)) \gets \texttt{vrf\_transcript}(\texttt{PedersenVrf}, \overline{io}, ad)\)
\item
  \(c \gets \texttt{challenge}([\bar{Y}, R, O_k], T)\)
\item
  \(\theta_0 \gets \top \text{ if } O_k + c \cdot O_m = s \cdot I_m \text{ else } \bot\)
\item
  \(\theta_1 \gets \top \text{ if } R + c \cdot \bar{Y} = s \cdot G + s_b \cdot B \text{ else } \bot\)
\item
  \(\theta = \theta_0 \land \theta_1\)
\end{enumerate}

Note: no public key appears in the verify inputs -- verification uses
the committed key \(\bar{Y}\) from the proof.

\subsection{4.3. Unblinding}\label{unblinding}

To link a Pedersen VRF proof to a specific public key, the prover
reveals the blinding factor \(b\) and the verifier checks:

\[\bar{Y} = Y + b \cdot B\]

where \(Y \in \bold{G}\setminus \{\mathcal{O}\}\) is the claimed public
key. The verifier MUST validate \(Y\) before accepting the association.

\subsection{4.4. Batch Verify}\label{batch-verify-1}

Multiple Pedersen VRF proofs can be verified together by combining the
individual verification equations with random weights (Schwartz-Zippel
lemma). Each proof contributes two equations (VRF correctness and
Pedersen commitment correctness), each weighted by an independent random
scalar.

\textbf{Input}:

\begin{itemize}
\tightlist
\item
  For \(j = 0, \ldots, N-1\): a tuple \((\overline{io}_j, ad_j, \pi_j)\)
  where:

  \begin{itemize}
  \tightlist
  \item
    \(\overline{io}_j \in (\bold{G}\times \bold{G})^{M_j}\): VRF
    input/output pairs.
  \item
    \(ad_j \in \bold{\Sigma}^*\): Additional data octet-string.
  \item
    \(\pi_j = (\bar{Y}_j, R_j, O_{k,j}, s_j, s_{b,j}) \in (\bold{G}, \bold{G}, \bold{G}, \bold{F}, \bold{F})\):
    Pedersen proof.
  \end{itemize}
\end{itemize}

\textbf{Output}:

\begin{itemize}
\tightlist
\item
  \(\theta \in \{ \top, \bot \}\): \(\top\) if all proofs verify,
  \(\bot\) otherwise.
\end{itemize}

\textbf{Steps}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  For each proof \(j\):

  \begin{enumerate}
  \def\labelenumii{\alph{enumii}.}
  \tightlist
  \item
    Validate \(\bar{Y}_j\), \(R_j\), \(O_{k,j}\), and all
    \(I_{j,i}, O_{j,i}\) \(\in \bold{G}\setminus \{\mathcal{O}\}\),
    output \(\bot\) if any is invalid or the identity.
  \item
    \((T_j, (I_{m,j}, O_{m,j})) \gets \texttt{vrf\_transcript}(\texttt{PedersenVrf}, \overline{io}_j, ad_j)\)
  \item
    \(c_j \gets \texttt{challenge}([\bar{Y}_j, R_j, O_{k,j}], T_j)\)
  \end{enumerate}
\item
  Derive random weights:

  \begin{enumerate}
  \def\labelenumii{\alph{enumii}.}
  \tightlist
  \item
    \(T_w \gets \texttt{new\_transcript}()\)
  \item
    \(T_w.\texttt{absorb}(\texttt{"pedersen-batch"})\)
  \item
    For each \(j\):
    \(T_w.\texttt{absorb}(\texttt{enc\_scalar}(c_j) \;\Vert\; \texttt{enc\_scalar}(s_j) \;\Vert\; \texttt{enc\_scalar}(s_{b,j}))\)
  \end{enumerate}
\item
  Check the combined equations:
  \[\sum_{j=0}^{N-1} t_j \cdot (O_{k,j} + c_j \cdot O_{m,j} - s_j \cdot I_{m,j}) + u_j \cdot (R_j + c_j \cdot \bar{Y}_j - s_j \cdot G - s_{b,j} \cdot B) = \mathcal{O}\]
  where \(buf_j \gets T_w.\texttt{squeeze}(32)\),
  \(t_j \gets \texttt{dec\_scalar\_mod}(buf_j[0..16])\),
  \(u_j \gets \texttt{dec\_scalar\_mod}(buf_j[16..32])\).
\end{enumerate}

\section{Appendix A. Recommendations}\label{appendix-a.-recommendations}

\subsection{A.1. Deterministic Secret Key Scalar
Generation}\label{a.1.-deterministic-secret-key-scalar-generation}

The following method derives a secret scalar from a 32-byte seed. It is
provided primarily for test vector generation and is not mandated by the
specification. Any secure method that produces uniformly distributed
scalars in \(\bold{F}\) is acceptable.

\textbf{Input}:

\begin{itemize}
\tightlist
\item
  \(seed \in \bold{\Sigma}^{32}\): seed octet-string.
\end{itemize}

\textbf{Output}:

\begin{itemize}
\tightlist
\item
  \(x \in \bold{F}\): secret key scalar.
\end{itemize}

\textbf{Steps}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \(i \gets 0\)
\item
  \(T \gets \texttt{new\_transcript}()\)
\item
  \(T.\texttt{absorb}(seed)\)
\item
  If \(i > 0\): \(T.\texttt{absorb}(i)\) where \(i\) is encoded as a
  single octet
\item
  \(d \gets \texttt{dec\_scalar\_mod}(seed)\)
\item
  \(x \gets \texttt{nonce}(d, T)\)
\item
  If \(x = 0\): increment \(i\) and go to step 2
\item
  Return \(x\)
\end{enumerate}

The seed is absorbed into the transcript and also passed as a scalar to
the \(\texttt{nonce}\) procedure (section 1.6.6), ensuring seed entropy
flows through both the transcript state and the secret scalar input
paths.

\subsection{A.2. Deterministic Blinding Factor
Generation}\label{a.2.-deterministic-blinding-factor-generation}

The following method generates the Pedersen VRF blinding factor
deterministically from the secret key and the VRF transcript state,
using the nonce function (section 1.6.6) with a distinct domain
separator. It is provided primarily for test vector generation;
implementations may use any method that produces a uniformly random
scalar in \(\bold{F}\).

\textbf{Linkability warning}: because \(b\) is derived deterministically
from \((x, T)\), two Pedersen VRF proofs with the same secret key, I/O
pairs, and additional data will produce the same blinding factor \(b\)
and therefore the same blinded public key
\(\bar{Y} = x \cdot G + b \cdot B\). An observer can detect that both
proofs originate from the same signer by comparing \(\bar{Y}\) values.
Applications that require unlinkability across repeated proofs on the
same inputs should generate \(b\) as a fresh uniformly random scalar
rather than using this deterministic method.

\textbf{Input}:

\begin{itemize}
\tightlist
\item
  \(x \in \bold{F}\): Secret scalar.
\item
  \(T\): Transcript state (from \(\texttt{vrf\_transcript}\)).
\end{itemize}

\textbf{Output}:

\begin{itemize}
\tightlist
\item
  \(b \in \bold{F}\): Blinding factor scalar.
\end{itemize}

\textbf{Steps}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \(T.\texttt{absorb}(\texttt{PedersenBlinding})\)
\item
  \(b \gets \texttt{nonce}(x, T)\)
\end{enumerate}

\section{Appendix B. HashTranscript
Construction}\label{appendix-b.-hashtranscript-construction}

Concrete instantiation of the transcript interface (section 1.6.1) using
SHA-512.

\textbf{Initialization}: \(\texttt{new\_transcript}()\) creates a fresh
SHA-512 state and feeds \(\texttt{suite\_id}\) into it.

\textbf{Absorb}: feeds raw bytes directly into the SHA-512 state.
Consecutive absorb calls are equivalent to a single absorb of the
concatenated data. This is safe because all protocol fields use
fixed-width encoding (\(\texttt{enc\_point}\): 32 bytes,
\(\texttt{enc\_scalar}\): 32 bytes, \(\texttt{enc\_32}\): 4 bytes,
domain tags: 1 byte) or explicit length prefixing (\(ad\) via
\(\texttt{enc\_32}(\texttt{len}(ad)) \;\Vert\; ad\)), so the byte stream
is unambiguous given the inputs agreed upon by both parties.

\textbf{Squeeze} (counter-mode XOF): on the first squeeze call, finalize
the SHA-512 state to obtain a 64-byte \(seed\). Then produce output
blocks:

\[block_i = \text{SHA-512}(seed \;\Vert\; \texttt{enc\_32}(i)) \quad \text{for } i = 0, 1, 2, \ldots\]

Each block yields 64 bytes. Output is read sequentially across blocks;
partial block state is preserved between squeeze calls.

\textbf{Fork}: duplicates the full internal state (including any partial
block position if squeezing has begun).

\section{Appendix C. Behavior with Zero I/O
Pairs}\label{appendix-c.-behavior-with-zero-io-pairs}

When \(n = 0\), the \(\texttt{vrf\_transcript}\) procedure (section
1.6.5) sets the merged pair to the identity:
\((I_m, O_m) = (\mathcal{O}, \mathcal{O})\). This causes the
VRF-specific verification checks to become trivially satisfied, since
any scalar multiplication with \(\mathcal{O}\) yields \(\mathcal{O}\).
The Schnorr proof-of-knowledge component, however, remains sound: a
valid proof still requires knowledge of the secret key \(x\).

The per-scheme behavior is as follows:

\begin{itemize}
\item
  \textbf{IETF VRF}: The verifier computes \(U = s \cdot G - c \cdot Y\)
  (non-trivial) and
  \(V = s \cdot \mathcal{O} - c \cdot \mathcal{O} = \mathcal{O}\)
  (vacuous). The scheme degenerates to a Schnorr signature on the
  additional data \(ad\), proving knowledge of \(x\) for public key
  \(Y\).
\item
  \textbf{Thin VRF}: The Schnorr pair \((G, Y)\) is always prepended
  (section 3.1, step 2), so the internal pair count is at least 1
  regardless of the user-supplied \(n\). The scheme remains a
  well-formed Schnorr proof even with zero VRF pairs.
\item
  \textbf{Pedersen VRF}: The VRF output check
  \(O_k + c \cdot O_m = s \cdot I_m\) is vacuously satisfied
  (\(\mathcal{O} = \mathcal{O}\)), but the commitment check
  \(R + c \cdot \bar{Y} = s \cdot G + s_b \cdot B\) still proves
  knowledge of the Pedersen commitment opening \((x, b)\).
\end{itemize}

No VRF output can be derived when \(n = 0\), since there are no output
points to hash.
